# 01_Blueprint_Foundation.md

### **1.0 Project Scale & Directives for Architects**

- **Classification:** Large-scale.
  - The workflow spans research, planning, code generation, review, and deployment stages that traditionally require several specialized teams.
  - Integrations include GitHub, Linear, Graphite, CodeMachine, and BYO agent endpoints, demanding adapter-driven complexity.
  - Deterministic resumability, rate-limit governance, and artifact governance increase the operational footprint beyond medium initiatives.
- **Rationale:** The specification targets multi-quarter evolution, repository agnosticism, and strict compliance with external APIs, all indicative of a large platform rather than a departmental tool.
  - Human and AI consumers both rely on machine-actionable artifacts, implying rigorous schema contracts.
  - Local-first design plus homelab reliability requirements force deep attention to fault-tolerance beyond smaller projects.
  - GitHub/Linear rate-limit handling, API version pinning, and branch protection intelligence are reliability mandates commonly associated with mature platforms.
- **Core Directive for Architects:** Architect for longevity, modular replacement, and minimal blast radius.
  - Every subsystem MUST expose explicit interfaces; no implicit coupling through shared globals.
  - Strong Separation of Concerns is mandatory: orchestration, integration, and artifact management tiers cannot bleed responsibilities.
  - Avoid proto-enterprise bloat by focusing on deterministic CLI workflows even while preparing for enterprise adoption.
- **Priority Outcomes:** Deterministic artifacts, resumable execution, and provider neutrality.
  - State machine checkpoints MUST survive power loss and network interruptions.
  - Artifact bundles MUST read the same way across machines, enabling portable debugging.
  - Providers (OpenAI-compatible endpoints, local LLMs, or external agents) plug in via declarative capability manifests.
- **Collaboration Mandate:** Specialized architects must consume this blueprint verbatim.
  - Structural_Data Architect owns schemas/interfaces but cannot alter mandated contracts without change-control.
  - Behavior Architect designs flows that respect approvals, gating, and retry semantics spelled out here.
  - Ops_Docs Architect documents operational runbooks consistent with local-first, no-daemon assumptions.
- **Systemic Guardrails:** Canonicalize all integration assumptions.
  - Capability flags govern optional modules (e.g., Linear agents preview, auto-merge, deployment triggers).
  - Feature flags default to conservative behavior; opt-in is required for any risky automation.
  - Error categories (transient/permanent/human-action) appear uniformly across components.
- **Design Focus Areas:** Architects must emphasize these aspects during detailed design.
  - State machine clarity trumps speed; prefer explicit transitions over implicit heuristics.
  - Security posture is least-privilege; secrets never land in artifacts without redaction tokens.
  - Observability is self-contained: logs, metrics, and traces live within run directories by default.
- **Traceability Requirements:** Every architectural artifact must map inputs to outputs.
  - PRD goals must link to spec requirements which tie to ExecutionTasks and resulting code diffs.
  - Run directories include `trace.json` summarizing relationships for programmatic consumption.
  - Architects must document trace IDs in every contract to prevent orphaned work items.
- **Risk Posture:** Favor failure containment over maximum throughput.
  - When in doubt, pause and request human confirmation rather than pushing risky automation.
  - State corruption triggers automatic archival of the affected run and requires manual acknowledgement before resuming.
  - Rate-limit or credential failures degrade gracefully to prompt-only workflows without losing collected context.
- **Escalation Protocol:** Ambiguities affecting rate limits, credential handling, or resumability must be escalated before implementation.
  - Document escalations in RepoConfig `governance_notes`.
  - No code merges occur until ambiguity resolution is recorded.
  - Emergency overrides require Foundation Architect approval captured via signed artifact entry.

### **2.0 The "Standard Kit" (Mandatory Technology Stack)**

- **Architectural Style:** Modular layered orchestrator with pluggable adapters.
  - Layers: CLI presentation → Orchestration Core → Adapter Interfaces → Artifact Persistence.
  - Each layer only depends downward; dependency inversion via TypeScript interfaces.
  - Deterministic run directories serve as the persistence plane; no background services.
- **Frontend:** No graphical UI; CLI is the product surface.
  - Commands must follow POSIX semantics and provide non-interactive flags for automation.
  - Optional TUI prompts are acceptable only if they degrade gracefully to plain text editors.
  - Output formats: stdout logs, markdown artifacts, JSON exports.
- **CLI User Experience:** Predictable behavior across shells.
  - Commands return structured exit codes (0 success, 10 validation, 20 external API, 30 human action required).
  - Provide `--json` flag for machine-readable output on `status`, `resume`, `export`, and `plan`.
  - Interactive editors default to `$EDITOR`; fallback to simple text prompts if unset.
- **Backend Language/Framework:** Node.js + TypeScript CLI.
  - Use `oclif` (preferred) for multi-command scaffolding, plugin discovery, and testing harnesses.
  - Enforce `tsconfig.json` with `strict`, `noImplicitAny`, and `exactOptionalPropertyTypes`.
  - Lint with `eslint` + `@typescript-eslint` presets; format with `prettier`.
- **Database(s):** Local-first storage.
  - Primary store: structured files in `.ai-feature-pipeline/<feature_id>/`.
  - Optional SQLite index (via `better-sqlite3`) for fast querying of runs; must be optional.
  - No external DB dependencies; remote persistence may be layered later via adapters.
- **Cloud Platform:** None required during execution.
  - Integrations talk directly to GitHub/Linear SaaS endpoints using HTTPS.
  - Architects must keep code cloud-agnostic so Docker/K8s adoption later requires zero refactoring.
  - Any remote callback mechanism must be optional and triggered via CLI commands, not persistent services.
- **Containerization:** Docker used for reproducible CI.
  - Provide `Dockerfile` referencing Node LTS v24 base image with deterministic package installs.
  - Container entrypoint must run CLI commands with mounted repository context.
  - Kubernetes deployment examples are out of scope but container should be stateless to support future orchestration.
- **Messaging/Queues:** In-process durable job queue.
  - Queue stored as append-only JSONL file with task metadata, status, retry policy, and timestamps.
  - Execution Engine reads from queue respecting dependencies defined in `plan.json`.
  - No reliance on RabbitMQ/Kafka; adapters may later target them but baseline must be file-based.
- **HTTP Layer:** Unified client module.
  - Built on `undici` (preferred) for performance and Node LTS compatibility.
  - Injects `Accept`, `X-GitHub-Api-Version`, `Authorization`, `Idempotency-Key`, and custom tracing headers.
  - Implements retry/backoff policies and surfaces structured error objects with response metadata.
- **Configuration Medium:** Repo-local configuration plus environment variables.
  - Canonical path: `.ai-feature-pipeline/config.json` with `zod`-validated schema.
  - Support `ai-feature init` to scaffold config and perform integration sanity checks.
  - Environment overrides follow the naming convention `AI_FEATURE_<SECTION>_<FIELD>`.
- **Testing Toolchain:** Comprehensive automation.
  - Use `vitest` for fast unit/integration tests with coverage reports stored under `.artifacts/tests`.
  - Provide fixture-based contract tests for adapters using recorded HTTP fixtures (`nock` or `msw`).
  - Add smoke-test scripts for CLI commands (`ai-feature start --prompt "test"`) running in CI.
- **Secret Management:** Environment-driven with redaction policies.
  - CLI warns when expected variables (`GITHUB_TOKEN`, `LINEAR_API_KEY`, `AGENT_ENDPOINT`) are missing.
  - Logger middleware scans outputs and replaces token substrings with `***REDACTED***`.
  - RepoConfig may reference OS keychain integration but default remains env vars.
- **State Machine Persistence:** Deterministic storage format.
  - `feature.json` records metadata, last_step, and last_error.
  - `plan.json` describes DAG edges; hashed to detect modifications.
  - `logs.ndjson` stores append-only events with monotonic timestamps.
- **Third-Party Integrations:** Adapter philosophy.
  - GitHub and Linear adapters share HTTP module and rate-limit ledger.
  - Graphite and CodeMachine integrations remain optional; capability flags determine availability.
  - Agent adapters (OpenAI, local LLM, external services) implement a standardized `AgentSession` contract.
- **Package Distribution:** Delivered as an npm package.
  - Publish under scoped name (e.g., `@ai-feature/pipeline`) with semantic versioning tied to blueprint revisions.
  - Provide installation instructions for both global (`npm install -g`) and project-local dependencies.
  - Release process emits changelog mapping features to blueprint version plus compatibility notes.

### **3.0 The "Rulebook" (Cross-Cutting Concerns)**

- **Feature Flag Strategy:** Config-first gating.
  - Flags stored in RepoConfig `feature_flags` object with default values.
  - CLI exposes `--enable-flag` overrides plus `.ai-feature-pipeline/flags.json` for persistent local toggles.
  - Incomplete user-facing behavior, risky git operations, and preview adapters MUST be behind flags disabled by default.
- **Observability (Logging, Metrics, Tracing):** Structured, deterministic telemetry.
  - Logging uses JSON lines with `timestamp`, `level`, `run_id`, `component`, `message`, and `context`.
  - Metrics exported as Prometheus textfiles (`metrics/prometheus.txt`) containing queue depth, retry counts, token usage, and API latency histograms.
  - OpenTelemetry tracing integrates via `@opentelemetry/sdk-trace-node`; default exporter is file-based `traces.json`, with optional OTLP endpoint when configured.
- **Security:** Defense-in-depth even in local runs.
  - Secrets never persist to disk unencrypted; when necessary, store hashed fingerprints for debugging.
  - Branch protection awareness is mandatory before merge attempts; CLI fails fast if required checks or reviews are missing.
  - Token scopes validated against required permissions (repo, pull_request, workflow) before operations commence.
- **Rate Limit Discipline:** Zero-tolerance for abuse endpoints.
  - HTTP client inspects `x-ratelimit-remaining`, `x-ratelimit-reset`, and `retry-after` headers, persisting them to `rate_limits.json`.
  - Secondary limit hits trigger exponential backoff with jitter and optional manual acknowledgement prompt for repeated failures.
  - Linear adapter enforces 1,500 requests/hour by maintaining sliding-window counters stored in run directory.
- **Resumability & Determinism:** Idempotence by design.
  - Each step stores input artifact hashes; unchanged inputs skip execution.
  - Crash recovery relies on file locks plus monotonic sequence numbers to avoid duplicate writes.
  - Resume command reads `last_step`, `last_error`, and replays queue entries until failure point.
- **Human-in-the-Loop Enforcement:** Explicit approvals.
  - Gate transitions: `research→PRD`, `PRD→Spec`, `Spec→Code`, `Code→PR`, `PR→Deploy`.
  - Approvals recorded in `approvals.json` with signer identity (user or agent) and timestamp.
  - Non-interactive mode accepts signed instruction bundles referencing exact artifact hashes.
- **Compliance & Auditability:** Artifacts as truth.
  - `ai-feature export` produces versioned bundles containing manifest, prompts, specs, diffs, and API transcripts.
  - Bundles include SHA digests for referenced files to detect tampering.
  - Audit metadata includes CLI version, Node version, OS fingerprint, and adapter versions.
- **Extensibility Contracts:** Explicit interfaces per capability.
  - Git provider interface: `getRepo`, `createBranch`, `commitPatch`, `createPullRequest`, `requestReviewers`, `getStatusChecks`, `merge`.
  - Ticket provider interface: `fetchIssue`, `snapshotIssue`, `updateIssue`, `postComment`.
  - Agent provider interface: `planTasks`, `generatePRD`, `generateSpec`, `proposePatch`, `summarizeDiff`.
- **Configuration Governance:** Schema discipline.
  - Every `config.json` stores `schema_version`; migrations documented and scripted.
  - CLI refuses to run if config schema is newer than CLI version unless `--force` is explicitly approved.
  - Change-control entries recorded in `config_history.json` to keep traceable lineage.
- **Data Provenance & Privacy:** Source tracking requirements.
  - Context entries store origin paths, commit SHAs, and capture timestamps.
  - Sensitive files flagged via RepoConfig `constraints.must_not_touch_paths`; aggregator redacts them from agent prompts.
  - Export bundles annotate every item with provenance metadata to satisfy auditing.
- **Error Taxonomy:** Consistent remediation guidance.
  - `Transient` errors (HTTP 429, network resets) trigger retries with exponential backoff up to configured attempts.
  - `Permanent` errors (validation failures, missing config) surface immediately with actionable instructions.
  - `Human-action-required` errors pause the run until an operator updates artifacts or approvals; resume picks up after acknowledgement.
- **Documentation Discipline:** Ops_Docs outputs follow template.
  - Every command includes synopsis, required inputs, side effects, and rollback guidance.
  - Rate-limit behavior documentation maps HTTP headers to CLI messaging for transparency.
  - Security documentation enumerates required GitHub PAT scopes and Linear API permissions.

### **4.0 The "Blueprint" (Core Components & Boundaries)**

- **System Overview:** The CLI orchestrator ingests prompts, tickets, or specs, harvests repo context, drafts PRDs/specs, plans execution, coordinates AI/human work, manages GitHub/Linear interactions, and delivers deterministic artifacts and deployment triggers while persisting every step locally for auditability.
  - Pipeline stages: Initialization → Context Gathering → Research → PRD Draft → Spec Draft → Task Planning → Execution → Validation → PR Automation → Deployment Trigger → Export.
  - Each stage has clear inputs/outputs and writes progress markers consumed by the resume engine.
  - Feature runs are isolated; cross-run communication happens only via RepoConfig and shared adapters.
- **Core Architectural Principle:** Strict Separation of Concerns.
  - Orchestration Core manipulates state machines; it never performs API calls directly.
  - Adapter layer owns all external communication, including authentication and rate-limit handling.
  - Artifact services own disk layouts, hashing, and bundle assembly; other modules interact via defined interfaces.

- **Key Components/Services:**
  - **CLI Orchestrator:** Command parser, lifecycle controller, and approval gatekeeper.
    - Routes commands to orchestrator workflows (init, start, status, resume, pr, deploy, export).
    - Maintains in-memory run context referencing disk artifacts.
    - Exposes hooks for telemetry and user prompts.
  - **RepoConfig Manager:** Configuration discovery and validation.
    - Detects git root, scaffolds `.ai-feature-pipeline/config.json`, and stores integration settings.
    - Performs sanity checks (GitHub repo fetch, Linear viewer query, Node version validation).
    - Publishes read-only config snapshot to other modules.
  - **Run Directory Manager:** Deterministic artifact handler.
    - Generates feature IDs (ULID/UUIDv7) and directory structures.
    - Manages file locks to prevent concurrent writes.
    - Ensures cross-platform path normalization and cleanup on failure.
  - **Context Aggregator:** Repository intelligence engine.
    - Collects README, docs, configured `context_paths`, and git history diffs.
    - Summarizes large files with token budgeting and records hash manifest.
    - Caches results to avoid re-gathering unchanged context.
  - **Research Coordinator:** Optional module generating ResearchTasks from unknowns.
    - Assigns tasks to agents or humans with objectives and cache keys.
    - Stores results in markdown plus structured metadata.
    - Marks stale tasks for refresh based on freshness requirements.
  - **PRD Authoring Engine:** Drafting subsystem.
    - Uses agent adapters to propose PRDs or falls back to templates.
    - Maintains review loop capturing edits and approvals.
    - Maps PRD goals to spec requirements for traceability.
  - **Specification Composer:** Engineering specification generator.
    - Converts PRD + research into structured spec with constraints, test plan, rollout plan, and risks.
    - Identifies unknowns requiring ResearchTasks.
    - Ensures spec references file paths, tests, and rollout toggles explicitly.
  - **Task Planner:** Dependency graph builder.
    - Breaks spec into ExecutionTasks with `depends_on` edges and capability tags.
    - Applies heuristics to group safe parallel tasks while honoring gating steps.
    - Writes `plan.json` consumed by Execution Engine.
  - **Execution Engine:** Patch-based executor.
    - Applies allowlisted diffs using `git apply` dry runs before committing.
    - Runs validations (lint/test/build) defined in config.
    - Records command outputs, statuses, and artifacts in logs.
  - **GitHub Adapter:** Git integration boundary.
    - Provides repo info, branch creation, PR creation, reviewer requests, merge operations, and status-check introspection.
    - Implements rate-limit aware HTTP calls with required headers.
    - Supports auto-merge enabling when repository settings allow it.
  - **Linear Adapter:** Ticket integration boundary.
    - Fetches issue metadata, snapshots payloads, respects rate limits, and handles Developer Preview agents via experimental flags.
    - Supports offline cache fallback when rate limited.
    - Updates Linear issues with status summaries when allowed.
  - **Agent Adapter Layer:** Provider abstraction.
    - Supports OpenAI-compatible endpoints, local models, or third-party agent services.
    - Negotiates capabilities (context window, tools, streaming) via manifest files.
    - Handles retries, cost estimation, and deterministic prompting strategies.
  - **Artifact Bundle Service:** Export and audit module.
    - Packages prompts, tickets, context lists, PRD/spec, plan, logs, diffs, rate-limit ledger, and PR info.
    - Supports `ai-feature export --format json|md`.
    - Validates completeness before marking run as archival-ready.
  - **Observability Hub:** Telemetry aggregator.
    - Collects logs/metrics/traces from all modules.
    - Applies redaction filters and writes to run directory plus optional remote sink.
    - Emits alerts when retries exceed thresholds or secrets appear in logs.
  - **Security & Credential Vault:** Secret governance.
    - Validates tokens for minimum scopes and expiration.
    - Provides token masking utilities to other modules.
    - Tracks credential usage to aid rotation policies.
  - **Deployment Trigger Module:** Merge/deploy automation.
    - Checks status checks, reviews, and branch protections prior to merge attempts.
    - Optionally enables auto-merge or triggers GitHub Actions workflow dispatch for deploy.
    - Records deployment results and status check evidence in `deployment.json`.
  - **Validation Command Registry:** Execution policy catalog.
    - Stores configured validation commands (lint, test, typecheck, build) plus required environment variables.
    - Determines which commands are mandatory per feature using RepoConfig metadata.
    - Provides dry-run capability to preview commands before execution.
  - **Resume Coordinator:** State recovery brain.
    - Reads `last_step`, `last_error`, queue files, and approval status to determine safe resumption point.
    - Validates artifact integrity via hashes before restarting execution.
    - Offers diagnostic summaries when resume is blocked by manual intervention requirements.

### **5.0 The "Contract" (API & Data Definitions)**

- **Primary API Style:** RESTful JSON contracts for external interactions; strict TypeScript interfaces for internal modules.
  - OpenAPI 3.1 specs describe optional HTTP callbacks (e.g., watchers) and CLI HTTP interactions.
  - Internal modules exchange data via typed interfaces validated by `zod`.
  - Schema versions are embedded in artifacts to ensure compatibility across CLI versions.
- **API Endpoint Catalog:** Canonical entry points for automation.
  - `POST /runs` initializes a feature when triggered via future services; accepts prompt or ticket reference.
  - `GET /runs/{id}` returns summarized state for dashboards or remote orchestrators.
  - `POST /runs/{id}/resume` triggers resumptions via signed instructions, mirroring CLI behavior.
  - `POST /runs/{id}/export` generates bundle and returns artifact path; used by optional remote tooling.
  - Even when HTTP endpoints are unused locally, their contracts govern CLI JSON outputs for parity.

- **Data Model - Core Entities:**
  - **Feature:**
    - Attributes: `id` (ULID/UUIDv7), `title`, `description`, `source`, `status`, `repo`, `working_branch`, `artifacts`, `external_links`, `acceptance_criteria`, `constraints`, `telemetry`.
    - Behaviors: state transitions (`draft`, `in_progress`, `review`, `done`, `deployed`) validated via finite-state machine.
    - Contracts: referenced by CLI commands and adapters; immutable IDs; stored in `feature.json`.
  - **RepoConfig:**
    - Attributes: `id`, `repo_url`, `default_branch`, `tool_integrations`, `context_paths`, `project_leads`, `github`, `linear`, `runtime`, `safety`, `feature_flags`, `governance_notes`.
    - Behaviors: versioned schema migrations stored alongside config with `schema_version`.
    - Contracts: `ai-feature init` writes baseline; `ai-feature status` shows resolved configuration and validation hints.
  - **RunArtifact:**
    - Attributes: `feature_id`, `prd_path`, `spec_path`, `plan_path`, `run_log_path`, `bundle_path`, `hash_manifest`.
    - Behaviors: ensures deterministic file naming and allows quick resumability checks.
    - Contracts: Execution Engine and Export service interact via path registry only; no hard-coded globs.
  - **ContextDocument:**
    - Attributes: `id`, `feature_id`, `path`, `sha`, `summary`, `token_cost`, `retrieved_at`, `source_kind`.
    - Behaviors: summarization metadata indicates compression level; redaction applied if secrets encountered.
    - Contracts: Context aggregator writes; agent adapters consume referencing IDs; Behavior Architect enforces budgets.
  - **ResearchTask:**
    - Attributes: `id`, `feature_id`, `title`, `objective`, `sources`, `status`, `result`, `assigned_to`, `cache_key`, `freshness_required`, `created_at`.
    - Behaviors: tasks transition from `pending` to `complete`; stale detection triggers re-run.
    - Contracts: Behavior Architect ensures tasks integrate into planning/resume flow; Ops_Docs documents operator expectations.
  - **Specification:**
    - Attributes: `id`, `feature_id`, `content`, `reviewer`, `status`, `created_at`, `change_log`, `test_plan`, `rollout_plan`, `risks`, `unknowns`.
    - Behaviors: review gate enforces acceptance before Execution Engine runs.
    - Contracts: Structural_Data Architect uses this schema when defining spec parsing logic and diff generation.
  - **ExecutionTask:**
    - Attributes: `id`, `feature_id`, `type`, `status`, `depends_on`, `retry_policy`, `assigned_agent`, `output`, `logs`, `created_at`.
    - Behaviors: tasks queue with dependency resolution; `retry_policy` includes `max_attempts`, `backoff`, and `retryable_errors`.
    - Contracts: Execution Engine updates statuses; Behavior Architect defines transitions; Ops_Docs documents human override hooks.
  - **IntegrationCredential:**
    - Attributes: `id`, `provider`, `auth_method`, `scope`, `expires_at`, `redaction_token`, `capability_flags`.
    - Behaviors: CLI validates scopes and warns before expiration; revalidation occurs before each API call burst.
    - Contracts: Security vault stores metadata but not raw tokens; logs reference `redaction_token` only.
  - **RateLimitEnvelope:**
    - Attributes: `provider`, `limit`, `remaining`, `reset_at`, `retry_after`, `backoff_attempts`, `last_error`, `last_checked`.
    - Behaviors: updated on every API response; persisted to allow resumable pacing.
    - Contracts: HTTP layer reads/writes to coordinate retries; Observability hub visualizes trends.
  - **DeploymentRecord:**
    - Attributes: `feature_id`, `pr_number`, `merge_sha`, `status_checks`, `required_reviews`, `auto_merge_enabled`, `deploy_job_url`, `completed_at`, `outcome_notes`.
    - Behaviors: monitors branch protection and deployment workflows; stores blocked reasons when applicable.
    - Contracts: Deployment module writes final state for audit artifacts; Ops_Docs documents remediation guidance.
  - **NotificationEvent:**
    - Attributes: `id`, `feature_id`, `channel`, `message`, `sent_at`, `delivery_status`, `audience`, `metadata`.
    - Behaviors: optional integration for email/slack/log updates; respects throttling policies.
    - Contracts: Behavior Architect ensures notifications respect feature flags and privacy constraints.
  - **AgentProviderCapability:**
    - Attributes: `provider`, `model_identifier`, `input_modality`, `tooling_support`, `cost_estimate`, `max_tokens`, `rate_limit_guidance`.
    - Behaviors: capabilities loaded from manifest files; CLI selects best provider per task and records selection rationale.
    - Contracts: Adapter interface consumes these attributes to configure agent sessions and fallback plans.

### **6.0 The "Safety Net" (Ambiguities & Assumptions)**

- **Identified Ambiguities:**
  - CLI scaffolding library and plugin strategy are unspecified beyond “standard POSIX CLI”.
    - Missing clarity on how plugins register commands or share context.
  - Optional watcher jobs for merged PR detection lack defined cadence or trigger mechanism.
    - No mention of scheduling approach (cron vs. manual) or locking semantics.
  - Bring-your-own-agent execution references multiple provider types without specifying manifest schema or negotiation workflow.
    - Ambiguous whether providers expose streaming APIs, tool use, or static prompts only.
  - Artifact bundle composition cites “diff summary” but leaves file layout undefined.
    - Unknown whether diff is textual patch, HTML summary, or JSON structure.
  - Deployment automation references GitHub Actions yet omits workflow ID patterns, required inputs, or secrets.
    - Branch protection handshake requirements for deploy gating are unclear.
  - Handling of inconsistent Node.js versions on user machines is unspecified.
    - No fallback plan for environments stuck on LTS-1 or lacking build tools.
  - Token rotation cadence and cache invalidation for Linear snapshots remain undefined.
    - Without guidance, stale data may drive incorrect planning decisions.
  - Graphite and CodeMachine integration depth is unspecified.
    - Requirements do not clarify whether these integrations are informational or control-plane capable.
  - Context summarization token budgets cite “large files” but not numeric thresholds.
    - Behavior of summarization when encountering binary files or generated assets is unknown.
  - Auto-merge enablement is described but not constrained.
    - Need clarity on who authorizes auto-merge per feature and how to roll back.
  - Agent cost accounting expectations are not stated.
    - Unclear how to report spend or enforce per-run budgets.
  - Local cache eviction policies for run directories are absent.
    - Without boundaries, storage usage may grow without operator awareness.

- **Governing Assumptions:**
  - CLI foundation standardizes on `oclif`.
    - Plugin registration uses `oclif` plugin mechanism; shared context is passed via dependency injection container.
  - Watcher behavior executes through `ai-feature observe` invoked via cron or CI schedule.
    - Command reads repo list from RepoConfig and runs idempotent checks; locking handled through file-based mutex.
  - Agent providers advertise capability manifests stored under `.ai-feature-pipeline/agents/<provider>.json`.
    - Manifest includes `models`, `max_tokens`, `tools_supported`, `rate_limits`, `feature_flags`, and `cost_per_token`.
  - Artifact bundle layout follows `bundle/manifest.json`, `bundle/docs/`, `bundle/logs/`, `bundle/diffs/`.
    - Diff summaries stored as unified patches plus machine-readable JSON metadata.
  - Deployment automation triggers GitHub Actions workflow specified in RepoConfig (`workflow_id`, `ref`, `inputs`).
    - Auto-merge is enabled only when `safety.require_human_approval_for_merge` is false and approvals are recorded.
  - Node.js version verification occurs during `ai-feature init`.
    - If runtime < v20, CLI aborts with instructions to install Node v24 or configure `nvm` alias; Ops_Docs documents remediation.
  - Token rotation is managed externally, but CLI revalidates scopes on each run.
    - Cached Linear issue snapshots store `retrieved_at`; stale snapshots older than 24h trigger refetch attempts with exponential backoff.
  - Graphite and CodeMachine integrations are telemetry/reporting in v1.
    - They consume exported artifacts but do not drive orchestration; adapters expose read-only sync endpoints.
  - Context summarization budgets default to 8k tokens per feature run.
    - Files larger than threshold are summarized with chunk metadata; binaries are hashed only.
  - Auto-merge authorization is recorded in `approvals.json`.
    - Disabling auto-merge occurs via CLI command `ai-feature pr disable-auto-merge <feature_id>` which updates DeploymentRecord.
  - Agent cost tracking writes spend estimates to `telemetry/costs.json`.
    - Resume operations reuse prior estimates unless new agent calls occur.
  - Run directories are tagged with expiration metadata; Ops_Docs architect defines cleanup script triggered via `ai-feature cleanup --before <date>`.
